## 1. Problem

Pi authors add LLM-callable tools with `pi.registerTool` (in-process TypeScript + TypeBox). next-code already has:

| Path | Shape | Gap |
|------|-------|-----|
| Core Rust tools | Built-in agent tools | Requires core PR |
| MCP servers | stdio / HTTP | Strong out-of-process bus; **Pi has no MCP** ([pi surfaces §2.10](../20260722-pi-extension-surfaces.md)) |
| Package / hooks | Lifecycle gates, not model tools | No package-declared *tool* yet |

Master open Q #2: *Should `[[tools]]` argv runners ship in Phase 3, or MCP-only until then?*

This file **resolves** that open Q and freezes the Phase 3 contract.

---

## 2. Resolved open Q (docs stance)

| Decision | Stance |
|----------|--------|
| **Default path for third-party tools** | **MCP-first** until argv tool bridge passes security review (D13) + allowlist + trust |
| **Phase 3 ships** | Package can declare tools as **MCP bridge** *and/or* **argv runners** behind policy gates |
| **When argv is allowed** | Only after D1 trust + D13 path-escape / argv-array / timeout / provenance rules are enforced or ticketed with owners |

**Rationale (one paragraph):** MCP is already next-code’s out-of-process tool bus and is a capability Pi lacks. Argv tools match herdr’s multilang spawn story and Pi’s *surface* (custom tool callable by the model), but they are **user-approved RCE** in the project tree — same threat class as command hooks. Shipping argv tools without D13 hardens the blast radius of “trust = RCE.” Prefer authors ship MCP servers in early Phase 3; unlock `runner.argv` when D13 exit criteria land.

**Changelog line (when frozen in product docs):**

> Phase 3 tools: MCP-first; `[[tools]]` argv gated behind trust + argv security (D13). Providers remain core/gateway — never guest-register.

## 3. Stance (frozen product rules)

1. **No in-process `registerTool` (Pi)** — no jiti, no Bun plugin tools, no resurrected QuickJS tool host (#49).
2. **No guest provider registration** — `pi.registerProvider` / stream adapters stay **core or gateway**, not package argv.
3. **Host permissions still apply** — package tools do not bypass DCG / permission mode / PreToolUse hooks.
4. **Face is paint-only** — tool *cards* remain Face’s fixed RenderBlock catalog; packages may supply **display hints** (Phase 4 D12), not custom Components ([Face limits §2.4 / §3](../20260722-face-customization-limits.md)).
5. **Multilang = argv or MCP process** — same language story as hooks/herdr; TypeScript is optional (`node dist/tool.js`), equal to Python/Bash.

## 4. Manifest shape (when enabled)

Extends D8 `next-code-plugin.toml` (filename still open at master level; shape is frozen here).

```toml
id = "example.deploy"
version = "0.1.0"
min_next_code_version = "…"

# --- Option A: MCP bridge (preferred early Phase 3) ---
[[tools]]
name = "deploy_status"
kind = "mcp"
# Points at package-local or referenced MCP server id already merged via [[resources]].mcp
mcp_server = "deploy-mcp"
mcp_tool = "status"          # optional rename; default = name

# --- Option B: argv runner (after D13) ---
[[tools]]
name = "deploy_status"
kind = "argv"
description = "Return deploy pipeline status for the current project."
input_schema = "schemas/deploy_status.json"   # JSON Schema draft relative to plugin root
# Optional: output_schema = "schemas/deploy_status_out.json"
runner = { argv = ["./bin/deploy-status"] }
timeout_ms = 15000
# Optional concurrency / execution mode
execution_mode = "sequential"   # sequential | parallel — maps to agent tool scheduling
```

### Field contract

| Field | Required | Notes |
|-------|----------|-------|
| `name` | Yes | LLM tool name; unique within enabled packages + core; collision → reject on link/enable |
| `kind` | Yes | `mcp` \| `argv` |
| `description` | argv: yes; mcp: optional override | Shown to model |
| `input_schema` | argv: yes | Path under plugin root; must be JSON Schema object |
| `output_schema` | No | If present, host may validate tool result before returning to model |
| `runner.argv` | argv: yes | **Array only** — no shell-string (D13 / herdr lesson) |
| `mcp_server` / `mcp_tool` | mcp: yes / optional | Server must be declared in package MCP resources |
| `timeout_ms` | No | Default from host settings; hard cap documented |
| `execution_mode` | No | Default `sequential` for argv (safer); MCP inherits server concurrency |

### Relative path rules

- `input_schema`, `runner.argv[i]` that look like paths: resolve under **plugin root** after canonicalization.
- Reject `..` escapes and absolute paths outside plugin root **except** allowlisted interpreters (D13).
- cwd for argv spawn = plugin root (herdr: `HERDR_PLUGIN_ROOT` pattern).

## 5. Runtime protocol (argv tools)

Mirror hooks stdio where possible so cookbook authors reuse skills ([`docs/HOOKS.md`](../../../HOOKS.md), herdr context JSON).

### 5.1 Invocation

1. Model selects tool `name` with JSON args matching `input_schema`.
2. Host runs PreToolUse hooks (existing) with tool name + input.
3. If allowed, host spawns `runner.argv` with:

| Env | Purpose |
|-----|---------|
| `NEXT_CODE_BIN_PATH` | Host callback (D6) |
| `NEXT_CODE_PLUGIN_ID` | Package id |
| `NEXT_CODE_PLUGIN_ROOT` | cwd |
| `NEXT_CODE_TOOL_NAME` | Tool name |
| `NEXT_CODE_TOOL_CALL_ID` | Correlation |
| `NEXT_CODE_TOOL_INPUT_JSON` | Optional duplicate of stdin for languages that prefer env |

4. **stdin:** one JSON object — validated args (+ optional host metadata envelope; keep envelope versioned).
5. **stdout:** one JSON object — tool result for the model (or `{ "content": …, "is_error": true }`).
6. **stderr:** diagnostics only; capped (herdr ~64 KiB lesson).
7. **exit codes:** `0` success; non-zero → tool error surfaced to model (do **not** invent shell-exit→allow/deny; that is hooks’ job).

### 5.2 Timeouts & concurrency

| Control | Default sketch |
|---------|----------------|
| Per-tool `timeout_ms` | 15_000 |
| Host hard max | e.g. 120_000 (settings) |
| Max concurrent argv tools | Align with hooks / herdr “max 32” class — start lower (e.g. 4–8) for tools |
| Kill on cancel | Abort spawn when session cancels tool (signal / kill tree — OS notes in D13) |

### 5.3 Provenance

Every spawn log line includes `plugin:<id> tool:<name> call:<id>` (D13). Face tool cards may show package id as a subtitle when ACP carries provenance (Phase 4 hints optional).

## 6. Runtime protocol (MCP bridge)

1. Package declares MCP in `[[resources]] mcp = […]` (D8) and merges into MCP registry (D4).
2. `[[tools]] kind = "mcp"` **aliases** a selected MCP tool into the agent’s active tool set with a stable `name`.
3. No second protocol — call goes through existing MCP client.
4. Enable/disable package must hide/show aliased tools (D2 enable-state).

**Why alias instead of “all MCP tools auto-exported”:** explicit model surface; avoids dumping entire server catalogs into every session; matches Pi’s deliberate `registerTool` / `setActiveTools` control.

| Concern | Pi | Herdr | next-code Phase 3 target |

**Steal:** Pi’s surface (“third party adds a tool without forking core”) + herdr argv/env spawn.  
**Avoid:** Pi in-process tools; herdr fire-and-forget-only (tools need result JSON).

## 8. Permission & policy layering

Order of enforcement (outer → inner):

```text
1. Package enabled? (D2)
2. Project trusted for executables? (D1) — argv only; MCP may have separate flag
3. Tool in active set / allowlist?
4. Host permission mode / DCG
5. PreToolUse hooks (can deny)
6. Spawn / MCP call
7. Optional output_schema validate
8. tool_result hooks (observe/sanitize)
```

**MCP trust:** inventory already notes optional `--require-mcp-trust`. Do not collapse MCP trust into project trust silently — document as separate control (D13 “Separate MCP trust”).

## 11. Validation rules (link / enable time)

| Check | Fail behavior |
|-------|---------------|
| Unknown `kind` | Reject package |
| argv without `input_schema` | Reject |
| Schema path escape / missing file | Reject |
| `runner.argv` empty or shell-string type | Reject |
| Name collision with core or another enabled package | Reject or warn+disable (prefer reject on enable) |
| `kind=mcp` but server missing | Reject |
| `min_next_code_version` too new | Reject (herdr pattern) |

## 12. Explicit non-goals (Phase 3)

- In-process TypeScript tools / Pi `ToolDefinition.execute` in host.
- Custom Face tool Components / overlays.
- Package-registered LLM providers / OAuth.
- WASM tool sandbox (Phase 5 only).
- Dynamic `registerTool` after session start without re-enable/reload (Pi does this; next-code v1 = manifest + reload — see D7).
- Treating “trusted project” as safe for unreviewed argv tools.
